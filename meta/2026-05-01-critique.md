# Self-critique 2026-05-01

The kb has 66 findings, 5 misses, 0 patterns. Big enough to critique
honestly. The picture is lopsided: the harness is excellent at
spec-review hunts and barely productive anywhere else. The single
biggest gap is that no one has run `hone abstract` in this entire
phase, so the obvious recurrent shapes are still loose findings.

## What's working

- **Spec-review hunting is the harness's superpower.** 55/66
  findings are `layer: spec` (53 against `docs/FWL_V02_SPEC.md`).
  That includes 12 contradictions, 18 ambiguities, 26 inconsistencies
  — most caught before code shipped. The `hone hunt` strategy logged
  5 hits in 5 runs (`meta/strategy_weights.json`); every one of those
  rounds drove a real spec edit. Without this pass Construct 3 would
  have shipped with a self-inverting dominator rule.
- **The few oracle-disagreement bugs the harness did find are
  exactly the right shape.** `finding/2026-05-01-v02-interpreter-tier2-v6-activation-helper-skips-function-body`
  and `finding/2026-05-01-v02-emitter-no-v4-parse-ok-gate-cross-family-disagreement`
  caught silent interpreter↔BPF drift on the dogfood program before
  soak. Those are exactly what a Class A oracle harness exists to
  catch.
- **Cross-document contradiction-hunting works.**
  `finding/2026-05-01-v02-pkt-spec-tcp6-decoded-proto-tcp-vs-fwl-activation-rule`
  and `finding/2026-05-01-v02-spec-strict-superset-vs-cross-family-pkt-proto`
  caught the FWL_V02_SPEC ↔ PKT_V02_SPEC interaction. The harness
  reads more than one document at a time.
- **Pkt-loader audits paid out.**
  `finding/2026-04-25-pkt-loader-state-rule-idx-not-validated`,
  `2026-04-25-pkt-loader-unknown-builder-fields`, and
  `2026-05-01-pkt-loader-accepts-noncanonical-ipv6-in-builder-fields`
  are an oracle-skew family that would otherwise have silently
  hidden divergences in every corpus case using those features.

## What's missing

  (`finding/2026-05-01-v02-interpreter-tier2-v6-activation-helper-skips-function-body`,

  root cause: a v0.1-vintage helper walks `program.rules` and

  `oracle-disagreement` findings that share the same Class A
  shape. These should be `patterns/helper-skips-function-body.md`

  `hone abstract` has never run successfully against this kb.

## Coverage gaps

- **Non-IP framing.** Zero findings exercise ARP, LLDP, MPLS,
  802.1Q with non-IP inner, or short Ethernet frames. The
  emitter's `v4_ok`/`v6_ok` gate fix was driven by IPv4 vs IPv6
  cross-family, not by non-IP. Spec finding
  `finding/2026-05-01-v02-spec-tier2-pkt-proto-no-dominator-rule`
  is the explicit hint that `p = pkt.proto` on a non-IP frame is
  undefined; nobody built a `.pkt` for it.
- **IPv6 extension headers.** No finding mentions hop-by-hop,
  routing, fragment, ESP, AH, or jumbograms. The v6 fixed-header
  bounds check is verified; the assumption that
  `next_header == TCP/UDP/ICMP6` matches the *first* byte after
  the fixed header (i.e., that no extension headers chain in
  between) is undocumented and untested. RFC 8200 says we have
  to walk the chain.
- **ICMPv6 / NDP.** One user-rule finding
  (`v6-internal-fw-ndp-dad-and-rs-dropped-against-stated-intent`).
  No finding probes whether the emitter's `icmp6` proto keyword
  matches every NDP packet shape (DAD, RS, RA, NS, NA, MLD)
  uniformly.
- **TCP flag combinations.** Findings list `pkt.tcp.syn` and
  `pkt.tcp.ack`. Nothing about `fin/rst/psh/urg/ece/cwr`, the
  reserved bits, malformed flag bytes, or TCP options. The L4
  bounds check addressed in
  `finding/2026-05-01-v02-emitter-no-v4-parse-ok-gate-cross-family-disagreement`
  was minimal-fix; we don't know the emitter handles a TCP header
  with options.
- **Geoip data path.** One harness finding (map population).
  Nothing on: bundle manifest parsing in `fd`, the dual-family
  split per call site, the JSON-vs-binary representation
  contradiction surfaced by
  `finding/2026-05-01-v02-spec-geoip-bundle-json-verbatim-vs-curated-subset`,
  the `MaxMind` ingestion in any tool that produces the bundle.
- **Tier 2 indented control flow.** The `FwlIndenter` postlex is
  gated on `DEF`. No finding probes: tab vs space mixing, blank
  lines inside indented blocks, EOF mid-indent, comments at
  arbitrary indent, deeply nested `if/elif/else` past whatever
  budget the indenter assumes.
- **Counter map readback.**
  `finding/2026-04-25-rate-limit-inversion-ssh-brute-force` notes
  the runner does not drain the per-CPU counter map. That same
  gap remains for Tier 2 `count` actions; no finding asserts
  pre/post-counter values match between interpreter and BPF.
- **Strict-superset under stress.** The v0.2 strict-superset
  promise has spec findings against it
  (`finding/2026-05-01-v02-spec-new-reserved-words-break-strict-superset-claim`,
  `finding/2026-05-01-v02-spec-icmp6-counter-name-breaks-strict-superset-carveout`,
  `finding/2026-05-01-v02-spec-strict-superset-vs-cross-family-pkt-proto`).
  Nothing replays the entire v0.1 corpus + every example through
  the v0.2 toolchain to demonstrate impl-level superset on a real
  v6 frame against a v0.1 program. The corpus regress at the gate
  signoff is a smoke test, not a strict-superset verification.

## Recommendations

1. **Run `hone abstract` now.** Three findings (count-slot keyerror,
   tier2-v6-helper, prior interpreter v6-surface gate) collapse into
   a single pattern `patterns/helper-skips-function-body.md`. Three
   oracle-skew runner/emitter/interpreter findings collapse into
   `patterns/v01-helper-not-extended-to-tier2.md`. With a pattern
   in hand, the next hunt round can grep every helper in
   `interpreter.py`, `emitter.py`, `analyzer.py`, `runner.py` for
   "walks `program.rules`" and prove for each one whether it should
   also walk `program.function.body`. This is the single highest-EV
   action.
2. **Add a `helper-walks-rules-only` strategy.** New `hone hunt`
   strategy whose prompt is: "list every function in
   `fwl/fwl/{interpreter,emitter,runner,analyzer,pkt}.py` that
   iterates `program.rules`. For each one, decide whether it must
   also descend into `program.function.body`. Produce a finding
   for any that doesn't and should." This generalises the pattern
   above into a recurring sweep rather than a one-shot abstract pass.
3. **Re-prompt the `oracle_divergence` strategy and rerun.** It
   ran once, scored zero, and the bugs it should have found
   instead came out of `hunt`. The strategy prompt is wrong. It
   should be: "run a v0.1-shaped program against a v6-builder
   `.pkt`, then a v6-activated program against a v4-builder
   `.pkt`, then `proto in [tcp, icmp6]` shapes, with each pkt's
   field set chosen so that interpreter and BPF would naturally
   diverge if any helper forgot a parse-path gate." Replay the
   three v6-activation findings as the seed corpus.
4. **Add a `non-ip-framing` strategy.** Pkt builders for ARP,
   LLDP, MPLS, short frames (ETH header truncated). Hypothesis:
   `p = pkt.proto` and `pkt.proto != tcp` on these inputs produce
   undefined behaviour per
   `finding/2026-05-01-v02-spec-tier2-pkt-proto-no-dominator-rule`,
   and the analyzer's dominator pass either rejects them or the
   emitter must clamp to a default. Verify against the just-fixed
   rule.
5. **Add an `extension-headers` strategy for IPv6.** Build `.pkt`
   cases with hop-by-hop, fragment, and routing extension
   headers between the v6 fixed header and the L4 header. Check
   the emitter still finds TCP/UDP/ICMPv6. This is the single
   most likely place a real packet trips the BPF parser in the
   48h soak.
6. **Make `mutate` and `transfer` actually productive.** Both
   strategies have 0 hits in 8 runs. Either retire them or seed
   them properly: feed `mutate` the existing 134-case corpus
   from `fwl/tests/corpus/` (especially `08_ipv6_fields/`,
   `09_geoip/`, `10_tier2_functions/`) with a bounded set of
   operators (flip a CIDR boundary by ±1, swap proto keyword to
   sibling family, drop a `pkt.` prefix, swap `==` for `in`).
   For `transfer`: take every fixed v0.1 finding's reproducer
   and rewrite as Tier 2 with an extra v6 condition. The
   v6-activation finding family is exactly what a transfer
   strategy should have produced and didn't.
7. **Probe the analyzer.** `analyzer.py` has 37 functions, 0
   findings. Write a `hone hunt` round whose only target is
   `f/fwl/fwl/analyzer.py`, with the spec dominator-rule
   findings 2026-05-01-v02-spec-tier2-* as seeds. Hypothesis:
   the polarity-aware dominator pass agrees with every spec
   bullet in `FWL_V02_SPEC.md:614-682`, plus the new
   `pkt.proto`-no-dominator hole, plus the negation/else gap.
   For each spec bullet, write a `.fw` case the analyzer should
   accept and a sibling it should reject.
8. **Probe the emitter beyond what oracle_divergence catches.**
   Static-inspection round on emitted C: hypothesis 1, every
   geoip LPM trie key honours `__be32`/`__be128` byte order on
   the ingest side as well as the lookup side; hypothesis 2,
   the v6 prelude `src_ip6_hi`/`src_ip6_lo` reads use
   `bpf_be64_to_cpu` consistently; hypothesis 3, the
   `__always_inline` lookup helper actually inlines under
   `clang -O2` (verify with `llvm-objdump`); hypothesis 4, every
   Tier 2 ActionStmt branch is reachable from the emitted C
   (no dead helpers).
9. **Tighten miss recording.** Update the hunt prompt so a hunt
   round's report includes a numbered hypothesis table even when
   no finding lands, in the shape of
   `miss/2026-04-25-analyzer-and-emitter-edge-cases`. The
   program-scoped misses (`internal_network`, `dogfood_v02`,
   `geoip_block`, `v6_internal`) hide what was *actually* tested
   inside each program; the next hunt round can't tell what's
   already been ruled out at hypothesis level.
10. **Audit `f/src/*.cc` and `fd`.** Six `.cc` files, zero
    findings. Before the 48 h dogfood soak starts is the time to
    run a static-inspection hunt on `bpf_loader.cc`,
    `slow_path.cc`, `engine.cc`. Hypothesis: the userland loader
    accepts a bundle whose geoip manifest disagrees with the
    BPF program's expected map shape (per
    `finding/2026-05-01-v02-spec-geoip-map-id-per-callsite-vs-per-callsite-family-pair`,
    the spec is contradictory; impl behaviour is unverified).
11. **Strict-superset regression suite.** Add a single
    `hone regress --strict-superset` mode that runs every v0.1
    corpus case with both a v4 and a v6 builder rebuild, asserts
    interpreter+BPF agree, and asserts the v6 case falls through
    for v0.1-shaped programs. This converts the strict-superset
    promise from "we believe it" to "the regress fails if it
    breaks." Right now the gate signoff cites 284/284 v0.1
    corpus pass, but only on v4 inputs.
