# 2026-05-01-v02-pkt-spec-tcp6-decoded-proto-tcp-vs-fwl-activation-rule

## Summary

`PKT_V02_SPEC.md` specifies the decoded-fields dict for the `tcp6`
(also `udp6`) builder with `"proto": "tcp"` unconditionally, plus the
explicit author comment "IPPROTO_TCP, identical for v4 and v6"
(`PKT_V02_SPEC.md:109-110`). `FWL_V02_SPEC.md` says the opposite
for v0.1-shaped programs: `pkt.proto == tcp` matches IPv6
`next_header == 6` only when the program activates the v6 parse
path (`FWL_V02_SPEC.md:161-168`, `:907-917`). The strict-superset
clause requires v0.1-only programs to keep v0.1's IPv4-only
semantics on v6 packets.

The two specs disagree on the same packet:

- The **AST interpreter**, fed the PKT_V02 decoded dict, reads
  `proto: "tcp"` and matches `pkt.proto == tcp` true on every
  `tcp6(...)` packet — including under a v0.1-shaped program.
- The **BPF runtime**, generated by the FWL_V02 emitter, never
  activates the v6 parse path for a v0.1-shaped program. On a v6
  TCP packet the EtherType-`0x0800` v4 parse fails and every rule
  falls through to the default action.

Same source, same packet, different verdict. Class A oracle drift.
Filed as spec-layer because the root cause is the cross-document
inconsistency: PKT_V02 hands the interpreter a decoded value that
contradicts FWL_V02's activation-conditional `pkt.proto` semantics.

The companion finding
`finding/2026-05-01-v02-spec-strict-superset-vs-cross-family-pkt-proto`
fixed the FWL_V02 side of this question (the proto-enum table is
now correctly conditional on v6-surface activation). The
corresponding PKT_V02 fix never landed, so the inconsistency lives
in the .pkt format spec instead of the language spec. The hone
agent that diffs FWL behaviour against PKT-derived oracle inputs
will surface this as a cross-oracle disagreement on the first v6
TCP test against a v0.1-shaped program.

## Locations

### PKT_V02 — decoded dict says `proto: "tcp"` unconditionally

`PKT_V02_SPEC.md:106-118`:

```python
# tcp6(src_ip="2001:db8::1", dst_port=80, syn=true) decodes to:
{
  "ether_type": 0x86DD,
  "proto": "tcp",          # IPPROTO_TCP, identical for v4 and v6
  "src_ip6": "2001:db8::1",
  "dst_ip6": "2001:db8::2",
  "src_port": 12345,
  "dst_port": 80,
  "syn": True,
  "ack": False,
}
```

The author's parenthetical comment "IPPROTO_TCP, identical for v4
and v6" is the assertion in question. It is correct at the wire
level (the IPv6 fixed-header `next_header` field for a TCP-bearing
v6 packet is byte 6, just as the IPv4 `protocol` field is). It is
**incorrect** at the language level under FWL_V02's activation rule.

### PKT_V02 cross-family-rule paragraph addresses src_ip but not proto

`PKT_V02_SPEC.md:120-124`:

> The IPv4 builders' decoded dict is unchanged (`src_ip` / `dst_ip`
> remain the v4 keys). Two builders never produce the same key set —
> so a rule referencing `pkt.src_ip` against an IPv6-builder packet
> finds the field absent and falls through, matching the spec's
> cross-family rule.

This paragraph **is** correctly aware of the cross-family
fall-through rule for L3 fields: an IPv4 program reading
`pkt.src_ip` against a `tcp6` packet finds the key missing in the
decoded dict and falls through. But the paragraph stops at L3
addresses; it does not extend the same logic to `pkt.proto`. The
result is asymmetric:

| Field | Cross-family handling in decoded dict | Outcome on a v6 packet under a v0.1 program |
|---|---|---|
| `pkt.src_ip` | Key absent (`tcp6` decodes `src_ip6`, not `src_ip`) | Interpreter falls through. Matches BPF (which falls through on EtherType). ✓ |
| `pkt.dst_ip` | Same | Same ✓ |
| `pkt.proto` | Key present, value `"tcp"` | Interpreter matches `pkt.proto == tcp`. BPF falls through (no v6 parse). ✗ |
| `pkt.src_port` | Key present, value `12345` | Interpreter matches `pkt.src_port == 12345`. BPF falls through. ✗ |
| `pkt.dst_port` | Same | Same ✗ |
| `pkt.tcp.syn` | Key present, value `True` | Interpreter matches `pkt.tcp.syn`. BPF falls through. ✗ |
| `pkt.tcp.ack` | Same | Same ✗ |

Six of the seven v0.1 `pkt` fields are decoded into the dict by the
v6 builders. Five of those six are subject to the activation rule
(everything except `pkt.proto` itself counts as a "v0.1-only
surface" — touching them does not flip the v6-parse switch). The
PKT_V02 decoded dict gives the interpreter the wire-level value for
all of them; under a v0.1-shaped program, that value contradicts
the BPF runtime's IPv4-only semantics.

### FWL_V02 — activation rule (post the strict-superset fix)

`FWL_V02_SPEC.md:161-168`:

> | Keyword | Matches when packet is | IPPROTO |
> |---|---|---|
> | `tcp` | IPv4 with `proto == 6` always; IPv6 with
>   `next_header == 6` (no ext hdrs) **only when the program
>   activates the v6 parse path** (see Compilation) | 6 |
> | `udp` | IPv4 with `proto == 17` always; IPv6 with
>   `next_header == 17` (no ext hdrs) **only when the program
>   activates the v6 parse path** | 17 |

`FWL_V02_SPEC.md:907-917`:

> For a program **touching IPv6 fields** (`pkt.src_ip6`,
> `pkt.dst_ip6`, `pkt.proto == icmp6`, or a `geoip(...)` whose LHS
> has type `ipv6`):
>
> - The analyser activates the IPv6 parse path in the emitter.
> ...
>
> For a program **not touching IPv6 fields** ...
>
> - The v6 parse path is **not** generated. Frames with EtherType
>   `0x86DD` (IPv6) fall through every rule, exactly as in v0.1, and
>   reach the default action.
> - `pkt.proto == tcp|udp` matches v4 frames only — same semantics
>   as v0.1.
> - This is the strict-superset guarantee: any v0.1 program compiled
>   by the v0.2 toolchain has identical packet behaviour to v0.1.

So the FWL spec is unambiguous: a v0.1-shaped program against a v6
TCP packet returns the default action, not a `pkt.proto == tcp`
match.

## Concrete .pkt that fires the divergence

```yaml
name: "v0.1-shaped program against tcp6 packet"
source_fw: |
  @xdp(eth0)
  drop if pkt.proto == tcp
  default allow

test_packet:
  builder: tcp6(src_ip="2001:db8::1", dst_port=80)

expected:
  compiles: true
  bpf_action: allow   # FWL_V02 :161-168 / :907-917 — v6 parse
                      # path inactive, packet falls through, default
                      # allow.
```

Three oracles run this case:

| Oracle | Implementation reads | Verdict |
|---|---|---|
| Spec | FWL_V02 :161-168 / :907-917 | `allow` |
| AST interpreter | PKT_V02 :109-110 — `proto: "tcp"` matches `pkt.proto == tcp` | `drop` |
| BPF runtime | Emitter follows :907-917 — no v6 parse path generated | `allow` |

The BPF and spec oracles agree; the interpreter disagrees because
PKT_V02 hands it the wrong decoded value.

This is the canonical Class A "spec-bug-disguised-as-implementation-bug"
shape: the implementer who looks at the corpus result will see
`interp:drop ≠ bpf:allow` and chase the interpreter as a bug. The
interpreter is faithfully reading the dict that PKT_V02 specifies.
The actual bug is at the spec level — the dict is wrong for
v0.1-shaped programs.

## Same shape for `udp6` and `icmp6`

The decoded dict template at `PKT_V02_SPEC.md:106-118` is for `tcp6`
specifically, but the same pattern applies to `udp6` (`proto:
"udp"`) and `icmp6` (`proto: "icmp6"`). For `udp6` the same Class A
divergence happens against any v0.1 program with `pkt.proto == udp`
or `pkt.dst_port == ...` clauses. For `icmp6` the activation rule
flips the other direction — a v0.1-shaped program **never** matches
`pkt.proto == icmp6` because v0.1 has no `icmp6` keyword, so an
`icmp6` builder against a v0.1 program falls through both interp
and BPF (no contradiction). The hazard is concentrated on the `tcp`
and `udp` proto values, which exist in v0.1.

## Why this matters

The strict-superset clause is the load-bearing guarantee for
Phase 2's regression methodology
(`planning/PHASE_2_PLAN.md`: "the v0.1 corpus is the regression
oracle for every Phase 2 PR"). The FWL_V02 side of the strict
superset was repaired by the related finding above. The PKT_V02
side was not, so:

1. As soon as the v0.2 corpus adds a v6 packet against any v0.1
   program — and the v0.2 dogfood plan includes this — the
   interpreter and BPF disagree. The hone regression run flags the
   case as a bug.
2. The implementer cannot fix the interpreter without contradicting
   PKT_V02; cannot fix the BPF emitter without contradicting
   FWL_V02. The fix has to be a spec change.
3. Worse, the misleading `# IPPROTO_TCP, identical for v4 and v6`
   comment is the kind of thing readers cite when arguing the
   interpreter is correct. The spec actively misleads.

## Proposed fix

Two paragraphs to edit:

### 1. PKT_V02 — make the v6-builder decoded dict context-aware

Replace `PKT_V02_SPEC.md:106-118` with:

```python
# tcp6(src_ip="2001:db8::1", dst_port=80, syn=true) decodes to:
{
  "ether_type": 0x86DD,
  "proto_byte": 6,         # wire-level IPPROTO_TCP value
  "src_ip6": "2001:db8::1",
  "dst_ip6": "2001:db8::2",
  "src_port_byte": 12345,
  "dst_port_byte": 80,
  "syn_byte": True,
  "ack_byte": False,
}
```

Add a paragraph: "The decoded dict carries the wire-level field
values under `_byte` suffixes; reading them through `pkt.<field>`
in the interpreter is governed by the v6-surface-activation rule
in `FWL_V02_SPEC.md:907-917`. A program that does not activate the
v6 parse path treats every `_byte` field on a v6 packet as
unreadable, identical to BPF runtime behaviour."

(The `_byte` rename is one option; the spec could equivalently keep
the existing keys and add a paragraph telling the interpreter
implementer to apply the activation rule. The rename makes the
contract harder to misread.)

### 2. PKT_V02 — extend the cross-family-rule paragraph

Replace `PKT_V02_SPEC.md:120-124` with:

> The IPv4 builders' decoded dict is unchanged (`src_ip` / `dst_ip`
> remain the v4 keys). Two builders never produce the same v4-key
> set — so a rule referencing `pkt.src_ip` against an IPv6-builder
> packet finds the field absent and falls through. The other six
> v0.1 `pkt` fields (`pkt.proto`, `pkt.src_port`, `pkt.dst_port`,
> `pkt.tcp.syn`, `pkt.tcp.ack`, plus `pkt.dst_ip`) are also subject
> to the v6-surface-activation rule of `FWL_V02_SPEC.md:907-917`:
> a v0.1-shaped program treats every read of those fields against a
> v6 packet as a fall-through, exactly as the BPF runtime does (the
> v6 parse path is inactive). The interpreter is responsible for
> applying this rule; the decoded dict alone is not enough.

### 3. (Optional) Update the `tcp6` example in PKT_V02 to use a program that activates v6 surface

The example at `PKT_V02_SPEC.md:32-44` uses
`allow if pkt.src_ip6 in 2001:db8:cafe::/48` — which **does**
activate v6 surface. So the example is fine. But adding a Worked
example block titled "v0.1-shaped program against a v6 packet"
that runs through the activation rule explicitly would close the
documentation gap.

## Class

Class A (interpreter and BPF runtime will produce different
verdicts for the same case as soon as the v0.2 corpus exercises
v6 packets against v0.1-shaped programs) with a spec-layer root
cause. The PKT_V02 spec hands the interpreter a decoded value
that contradicts FWL_V02's activation-conditional `pkt.proto`,
`pkt.src_port`, `pkt.dst_port`, `pkt.tcp.syn`, and `pkt.tcp.ack`
semantics.
