# 2026-05-01-v02-interpreter-missing-v6-surface-activation-gate

## Summary

The AST interpreter (`fwl/fwl/interpreter.py`) reads the v0.2 v6
builder's decoded packet dict directly for v0.1-style fields
(`pkt.proto`, `pkt.src_port`, `pkt.dst_port`, `pkt.tcp.syn`,
`pkt.tcp.ack`, `pkt.src_ip`, `pkt.dst_ip`) regardless of whether the
program activates the v6 parse path. Per `PKT_V02_SPEC.md:122-127`
and `FWL_V02_SPEC.md` §"Compilation" (the v6-surface activation
rule), a **v0.1-shaped program** (one that does not touch
`pkt.src_ip6`, `pkt.dst_ip6`, the `icmp6` proto keyword, an IPv6
literal, or an IPv6 CIDR) must treat reads of those fields against a
v6-builder packet as **unreadable** — the v6 parse path is inactive
in the BPF emitter, so the rule must fall through.

The interpreter does not enforce this gate. Result: for any v0.1-
shaped FWL program tested against a `tcp6`/`udp6`/`icmp6` builder
packet whose decoded `proto`/`src_port`/`dst_port`/`syn`/`ack` would
satisfy a rule, the interpreter returns the rule's action while the
BPF runtime returns the default action (because the v0.1-shaped
emitter never populates those locals for a v6 frame).

## Hypothesis

`PKT_V02_SPEC.md:124-126` says explicitly:

> For a program **not touching IPv6 fields** (a v0.1-shaped
> program), the interpreter must treat every read of `pkt.proto`,
> `pkt.src_port`, `pkt.dst_port`, `pkt.tcp.syn`, `pkt.tcp.ack`,
> `pkt.src_ip`, `pkt.dst_ip` against a v6-builder packet as
> **unreadable** (the v6 parse path is inactive in the BPF emitter;
> the rule falls through, identical to BPF behaviour).

The interpreter has no notion of "v6 surface activation"
(`emitter._is_v6_active` is a private emitter-only helper). It
unconditionally calls `packet.get(<key>)` on the decoded dict, and
the v6 builder populates `proto`, `src_port`, `dst_port`, `syn`,
`ack` keys with the wire-level truth.

## Evidence

`corpus/from_hunt/2026-05-01_v6_internal_hunt/v01_shaped_vs_v6_packet.pkt`:

```yaml
source_fw: |
  @xdp(eth0)
  allow if pkt.proto == tcp and pkt.dst_port == 80
  default drop

test_packet:
  builder: tcp6(src_ip="2001:db8::1", dst_port=80)

expected:
  bpf_action: drop
```

Run output:

```
FAIL  v01_shaped_vs_v6_packet.pkt  (...)
      spec         pass
      interpreter  fail  -- expected XDP_DROP, got XDP_PASS
      bpf          skip  [...clang compile passed]
```

Inspecting the emitted BPF C for the same source confirms the
emitter omits any v6 parse path:

```c
// Generated by fwl. Do not edit.
...
SEC("xdp")
int fwl_prog(struct xdp_md *ctx) {
  __u8 proto = 0;
  __u16 dst_port = 0;
  ...
  if ((void *)(eth + 1) <= data_end) {
    if (eth->h_proto == bpf_htons(ETH_P_IP)) {
      // v4 parse only
    }
    // No v6 branch.
  }
  if (((proto == IPPROTO_TCP) && (dst_port == 80))) return XDP_PASS;
  return XDP_DROP;
}
```

For a v6 frame (EtherType `0x86DD`) `proto` and `dst_port` stay 0,
so the rule does not match and the program returns `XDP_DROP`. The
BPF oracle would report `drop`. The interpreter — reading
`packet["proto"]` = `"tcp"` and `packet["dst_port"]` = `80` directly —
returns `XDP_PASS`. The two oracles disagree on every v0.1-shaped
program whose rule's L4 conjunct happens to match the v6 builder's
defaults.

This is the canonical type-A bug (interpreter and BPF runtime
disagree on the same packet). The runtime test is currently
demonstrated only at the interpreter layer (the BPF run is skipped
for lack of CAP_BPF on this host) but the emitted C is structurally
unambiguous.

## Surface area

The bug applies to any v0.1-shaped FWL program — the entire v0.1
corpus, every dogfood `.fw` that doesn't touch v6 — when fed any
`tcp6`/`udp6`/`icmp6` packet whose v0.1-style fields satisfy a rule.
Concretely the failing reads are:

| Field | v6 builder source | Spec required behaviour for v0.1-shaped program |
|---|---|---|
| `pkt.proto` | decoded `proto` ("tcp"/"udp"/"icmp6") | unreadable |
| `pkt.src_port` | decoded `src_port` | unreadable |
| `pkt.dst_port` | decoded `dst_port` | unreadable |
| `pkt.tcp.syn` | decoded `syn` | unreadable |
| `pkt.tcp.ack` | decoded `ack` | unreadable |
| `pkt.src_ip` | (absent — v6 builder uses `src_ip6`) | already unreadable, no bug |
| `pkt.dst_ip` | (absent — v6 builder uses `dst_ip6`) | already unreadable, no bug |

The v4-only fields (`src_ip`, `dst_ip`) coincidentally behave
correctly because the v6 builder's decoded dict simply omits them.
The five fields above are the actual surface.

## Recommended fix

Either:

1. *Interpreter-side gate.* Add a v6-surface-activation check to
   `interpreter.evaluate` mirroring `emitter._is_v6_active`. When the
   program is v0.1-shaped and the packet's decoded dict carries
   `ether_type == 0x86DD`, treat reads of the five fields as `None`
   (so `_eval_comparison` falls through and the rule does not match).

2. *Builder-side gate.* Have `pkt._build_v6_packet` write decoded
   `proto`/`src_port`/`dst_port`/`syn`/`ack` only when the program
   activates v6. This couples builder to program, which is awkward
   — option 1 is cleaner.

The fix lives in `interpreter.py`. A `_v6_active(program)` helper
that the public `evaluate` calls once, paired with a
`packet["ether_type"] == 0x86DD` check at field-read time, closes
the gap.

The runner's BPF oracle should be unaffected — the emitter is
already correct (no v6 parse path for v0.1-shaped programs).

## Affects

- Every v0.1 corpus case that the operator regresses against a v6
  packet would silently misagree at the interpreter layer.
- Phase 2 dogfood `dogfood_v02.fw` is v6-active so unaffected.
- `examples/v6_internal.fw` is v6-active so unaffected.
- Phase 1 examples (`internal_network.fw`, `ssh_brute_force.fw`,
  `web_server_ddos.fw`) are all v0.1-shaped; running them against a
  `tcp6`/`udp6` packet would trip the bug.
